# photopipe

A local-first command-line pipeline for RAW photo libraries.

**Status:** Phase 1 complete — directory scanning, DuckDB catalog, EXIF extraction, content-addressable WebP preview cache, and full idempotency are working. Defect detection, deduplication, and review-tree generation are not yet implemented (Phases 2–7).

## Install

```sh
cargo install --path crates/cli
```

Or build from source:

```sh
cargo build --release
# binary at: target/release/photopipe
```

## Quick start

```sh
# Check system health and print effective config
photopipe doctor

# Scan a directory of RAW + JPG files
photopipe scan ~/Pictures/2024

# Run again — second pass skips all unchanged files (idempotent)
photopipe scan ~/Pictures/2024

# Show per-flag statistics (Phase 7)
photopipe stats
```

After a scan the catalog can be queried directly:

```sh
duckdb ~/.local/share/photopipe/catalog.duckdb \
  "SELECT path, file_format, content_hash FROM files LIMIT 10"

duckdb ~/.local/share/photopipe/catalog.duckdb \
  "SELECT f.path, e.camera_model, e.focal_length_mm, e.aperture
   FROM files f JOIN exif e ON e.file_id = f.id
   WHERE e.camera_model IS NOT NULL"
```

## Configuration

Default config path: `$XDG_CONFIG_HOME/photopipe/photopipe.toml` (falls back to `~/.config/photopipe/photopipe.toml`).

The file is optional — all settings have sensible defaults. Copy `photopipe.example.toml` as a starting point. Override the path with `--config <path>`.

Default data locations:

| Purpose | Default path |
|---------|-------------|
| Catalog DB | `$XDG_DATA_HOME/photopipe/catalog.duckdb` |
| Preview cache | `$XDG_CACHE_HOME/photopipe/previews/<2-char-bucket>/<hash>.webp` |

## How scanning works

1. `photopipe scan <dir>` walks the directory tree (via `walkdir`) in parallel (via `rayon`).
2. Each file is stat'd and checked against the catalog `(path, mtime_ns, size_bytes)`. Unchanged files are skipped — zero work on repeat runs.
3. New or changed files are hashed with xxh3-128, EXIF is read (rawler for RAW formats, kamadak-exif for JPEG), and a preview is extracted and encoded as lossless WebP into the content-addressable cache.
4. Results are written to the DuckDB catalog in batches (one transaction per batch of 64 files).

## Implementation notes and known deviations from spec

### `GENERATED BY DEFAULT AS IDENTITY` → sequences

The spec calls for DuckDB identity columns (`BIGINT GENERATED BY DEFAULT AS IDENTITY`). The bundled DuckDB version (1.1.x) does not support this syntax. The schema instead uses explicit sequences:

```sql
CREATE SEQUENCE seq_files_id START 1;
CREATE TABLE files (
    id BIGINT PRIMARY KEY DEFAULT nextval('seq_files_id'),
    ...
);
```

The behaviour is identical. This can be migrated to the identity syntax once the bundled DuckDB version is updated.

### WebP encoding is lossless

The `image` crate 0.25.x `WebPEncoder` only exposes a lossless encoder (`new_lossless()`). The `preview_quality` config value is accepted and stored for forward compatibility but is currently ignored — all previews are written as lossless WebP. Lossy encoding at the configured quality level will require either a newer `image` release that exposes quality-based WebP or switching to the dedicated `webp` crate.

## Supported RAW formats

`arw` (Sony), `cr3` / `cr2` (Canon), `nef` (Nikon), `raf` (Fujifilm), `rw2` (Panasonic), `dng` (Adobe DNG), `jpg` / `jpeg`.

Add or remove extensions via `ingest.extensions` in your config file.
